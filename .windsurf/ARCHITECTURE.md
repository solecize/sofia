## Canonical Work Structure

Each project in `corpus/works/<project>/` follows this structure:

```
corpus/works/<project>/
├── chapters/
│   ├── 01-chapter-one.md
│   ├── 02-chapter-two.md
│   └── ...
├── notes/
│   ├── notebook.md              # Working hub for this project
│   ├── 01-chapter-one-notes.md  # Notes for chapter 1
│   ├── 02-chapter-two-notes.md  # Notes for chapter 2
│   └── ...
├── reference/                   # Wiki-style reference (NOT prose)
│   ├── people/
│   ├── places/
│   ├── objects/
│   ├── events/
│   └── themes/
├── .sofia/
│   ├── work.json                # Work metadata
│   └── profile.md               # Work profile (genre, phase, etc.)
├── manuscript.md                # Table of contents / compiled view
└── orphans.md                   # Unplaced prose not yet in chapters
```

### Key Files

| File | Purpose |
|------|---------|
| `notebook.md` | Working hub with stats, links, quick reference for this project |
| `<chapter>-notes.md` | Summary and reference links for each chapter |
| `manuscript.md` | TOC linking to all chapters |
| `orphans.md` | Prose fragments not yet assigned to chapters |

### Chapter-Level Notes

Each chapter has a corresponding notes file:
- Chapter: `chapters/01-introduction.md`
- Notes: `notes/01-introduction-notes.md`

Chapter notes contain:
- Summary of the chapter
- Links to relevant reference entries
- Continuity notes
- TODO items for revision

## Corpus-Level Structure

```
corpus/
├── index.md          # Author dashboard (auto-generated)
├── incoming/         # Raw imports waiting to be processed
│   └── tutorial/     # Tutorial demo content (public domain texts)
└── works/            # Individual writing projects (user content)
```

Note: `corpus/works/` is empty by default. Demo works are created via the tutorial.

### Dashboard (`corpus/index.md`)

Auto-generated by `scripts/sofia-dashboard`. Shows:
- Works table (name, words, chapters, phase, last edit)
- Recent activity per work (3 most recent files)
- Links to each work's `notes/notebook.md`

Regenerated on file changes by SofiaMonitor.

## CLI Scripts

| Script | Purpose |
|--------|---------|
| `sofia-work` | Manuscript management (init, ingest, status, checkin, checkout) |
| `sofia-tutorial` | Interactive tutorial for new users |
| `sofia-dashboard` | Generate `corpus/index.md` dashboard |
| `sofia-wiki` | Entity extraction and reference management |
| `sofia-mx` | MX introspection commands |

### sofia-dashboard

```bash
./scripts/sofia-dashboard           # Generate corpus/index.md
./scripts/sofia-dashboard --json    # Output JSON (for app integration)
```

### sofia-tutorial

```bash
./scripts/sofia-tutorial            # Start interactive tutorial
./scripts/sofia-tutorial --resume   # Resume from saved state
./scripts/sofia-tutorial --reset    # Reset tutorial progress
./scripts/sofia-tutorial --install-cleanup  # Install cleanup script
```

Tutorial state stored in `~/.sofia/tutorial-state.json`.
Tutorial-created content tracked in `~/.sofia/tutorial-created.json`.
Cleanup script at `~/.sofia/tutorial-cleanup` (local only, not in repo).

## SofiaMonitor (macOS Menu Bar App)

Native SwiftUI app for file monitoring and quick access.

### Building

```bash
cd SofiaMonitor
./build.sh --run    # Build and run
```

**Command-line compilation notes:**
- `#Preview` macros must be commented out (require Xcode)
- Asset catalogs not available - use embedded resources
- Output filename must differ from directory name

### Key Components

| File | Purpose |
|------|---------|
| `SofiaMonitorApp.swift` | Main app, AppState, startTutorial() |
| `Views/MenuBarView.swift` | Menu bar dropdown UI |
| `Views/PreferencesView.swift` | Settings window |
| `Models/Environment.swift` | Environment model with paths |
| `Services/FileWatcher.swift` | FSEvents file monitoring |
| `Services/GitService.swift` | Git operations (commit, status) |

### File Monitoring Flow

1. FileWatcher monitors `corpus/works/` via FSEvents
2. On change, 2-second debounce
3. Auto-commit with message: `auto: workname filename @ timestamp`
4. Regenerate dashboard via `sofia-dashboard`

### Menu Bar Actions

- **Open Dashboard** → Opens `corpus/index.md` (uses document editor)
- **Start Tutorial...** → Launches `sofia-tutorial` in Terminal
- **Open in [Editor]** → Opens environment/work in selected editor

### Editor Routing

Files are routed to editors by extension:

| Category | Extensions | Setting |
|----------|------------|--------|
| Environment | Directories, `.swift`, `.sh`, `.json`, `.py`, `.js`, `.ts`, etc. | `@AppStorage("environmentEditor")` |
| Document | `.md`, `.txt`, `.rtf`, `.markdown` | `@AppStorage("documentEditor")` |

Options for each:
- Specific app (Windsurf, Cursor, VS Code / Typora, iA Writer, Obsidian)
- `"system"` → Uses `NSWorkspace.shared.open()` to respect Finder defaults

The Preferences → Editors tab shows "OS Default (AppName)" where AppName is queried from `NSWorkspace.shared.urlForApplication(toOpen:)`.

### Environment Locking

When lock is enabled via Preferences → Environments:

1. **Lock file created:** `.sofia/environment.lock`
   ```
   locked_path: /Users/you/Documents/sofia
   locked_at: 2026-03-29T15:00:00Z
   environment_name: sofia
   ```

2. **Rules updated:** Lock restriction appended to `.windsurfrules`:
   ```markdown
   <!-- SOFIA ENVIRONMENT LOCK -->
   ## Environment Restriction
   **CRITICAL:** This environment is LOCKED. You must NEVER access files outside:
   `/Users/you/Documents/sofia`
   <!-- END SOFIA ENVIRONMENT LOCK -->
   ```

3. **AI behavior:** Assistants should check for `.sofia/environment.lock` and respect the `locked_path` boundary.

### Writing Organization Mode

When enabled via Preferences → AI Assistant:

1. **Rules injected:** MX safety rules prepended to `.windsurfrules` (or `.cursorrules` / `.github/copilot-instructions.md`)
2. **Template embedded:** Rules are bundled in the app (no external file dependency)
3. **User rules preserved:** Existing rules wrapped in `<!-- USER RULES -->` markers
4. **Backup created:** Dated backup (e.g., `.windsurfrules.backup-20260329`)

Check status with: `sofia-mx status`

## Development Workflow

See `.windsurfrules` for the full rule set. Key points:

### Documentation Parity

Every code change must include corresponding documentation updates:

1. **Plan phase**: Flag both code AND doc changes
2. **Implementation**: Update docs alongside code
3. **Doc review checkpoint**: Verify docs match new code before commit
4. **Release**: Code + docs + CHANGELOG entry together

### Documents by Change Type

| Change Type | Check These Docs |
|-------------|------------------|
| New feature | README.md, documentation/, CHANGELOG.md |
| CLI change | documentation/sofia-*.md, CHANGELOG.md |
| Config change | documentation/sofia-monitor.md, ARCHITECTURE.md |
| Structure change | ARCHITECTURE.md, documentation/mx/*.md |
| Deprecation | CHANGELOG.md (with migration notes) |

## Branch Strategy

| Branch | Purpose |
|--------|---------|
| `main` | Development branch. All work happens here. |
| `public` | Release branch. GitHub default. Only updated after review. |

### Release Process (main → public)

1. **Verify no private content:**
   ```bash
   git ls-tree -r main --name-only | grep -E "corpus/works/|corpus/incoming/" | grep -v tutorial
   ```

2. **Review the diff:**
   ```bash
   git diff public main --stat
   git log public..main --oneline
   ```

3. **Get explicit user approval** before merging

4. **Merge and push:**
   ```bash
   git checkout public
   git merge main -m "Merge main into public: <summary>"
   git push origin public
   git checkout main
   ```

**IMPORTANT:** Never push to `public` without explicit user approval. The `public` branch is what GitHub displays by default.
